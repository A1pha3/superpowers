## 核心设计哲学

### 为什么叫"Superpowers"？

**Superpowers**（超能力）这个名字不是随意选择的。

```
普通人 → 普通开发者
超人 → 有超能力的开发者

Superpowers 把普通 AI 编码代理变成了"超级"代理
```

就像超能力需要正确使用才能发挥效果，Superpowers 的技能也需要遵循正确的工作流程。

## 四大核心原则

### 1. 测试驱动开发（Test-Driven Development）

**原则**：始终先写测试

**为什么重要**：

```
没有测试的代码就像没有安全网的走钢丝。
你可能成功，但失败时代价昂贵。
```

**传统方式的致命缺陷**：

| 问题 | 测试后写 | 测试先写（TDD） |
|------|---------|----------------|
| 验证 | 测试通过证明什么？ | 看到失败证明测试有效 |
| 覆盖 | 可能遗漏边界情况 | 边界情况在编写时发现 |
| 设计 | 测试适应实现 | 实现满足测试需求 |
| 回归 | 不敢改动 | 有测试保护 |

**铁律**：

```markdown
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST

没有失败的测试，就没有生产代码。
```

这不是建议，这是强制规则。

### 2. 系统化而非临时性（Systematic Over Ad-hoc）

**原则**：流程优于猜测

**AI 代理的常见问题**：

```
临时性方法：
问题 → 猜测原因 → 尝试修复 → 新问题 → 重复

系统化方法：
问题 → 系统分析 → 根本原因 → 目标修复 → 验证
```

**为什么临时性方法会失败**：

1. **认知偏差**：你倾向于在熟悉的领域寻找原因
2. **确认偏差**：你只看到支持你假设的证据
3. **沉没成本**：你坚持错误的路径因为已经投入时间

**系统化方法的优势**：

```
调试技能的四阶段流程：

根因追踪 → 找到真正原因
深度防御 → 多层验证
条件等待 → 稳定复现
修复验证 → 确保真正修复

每个阶段都有明确的输入和输出。
```

### 3. 降低复杂度（Complexity Reduction）

**原则**：简单性是主要目标

**YAGNI（You Aren't Gonna Need It）**：

```
每个功能都是：
- 需要写的代码
- 需要维护的代码
- 可能出错的代码
- 需要理解的代码

不要写你现在不需要的功能。
```

**复杂度的代价**：

| 复杂度来源 | 代价 |
|-----------|------|
| 不必要的功能 | 维护负担 |
| 过度抽象 | 理解困难 |
| 过早优化 | 浪费时间 |
| 多种选择 | 决策疲劳 |

**Superpowers 如何降低复杂度**：

1. **强制小任务**：2-5 分钟的任务难以过度设计
2. **单一职责**：每个技能做一件事
3. **渐进式设计**：验证后再继续

### 4. 证据优于声明（Evidence Over Claims）

**原则**：在宣布成功之前先验证

**常见的"成功"声明**：

```
❌ "我修复了这个 bug" → 但 bug 还在
❌ "测试通过了" → 但测试覆盖错误
❌ "功能完成了" → 但不符合需求
```

**Superpowers 要求的证据**：

| 声明 | 需要的证据 |
|------|-----------|
| "测试通过" | 输出显示绿色 |
| "Bug 修复" | 复现步骤显示不再出现 |
| "符合需求" | 与计划对照检查 |
| "代码清洁" | 代码审查通过 |

**verification-before-completion 技能**：

```markdown
## 完成前验证清单

- [ ] 问题能够在修复前复现
- [ ] 根本原因已识别
- [ ] 修复针对根本原因
- [ ] 测试覆盖了修复
- [ ] 问题在修复后不再存在
- [ ] 没有引入新问题
```

## 关键设计决策

### 决策 1：强制工作流 vs 建议

**选择**：强制工作流

**权衡**：

| 方面 | 建议 | 强制 |
|------|------|------|
| 灵活性 | 高 | 低 |
| 一致性 | 低 | 高 |
| 学习曲线 | 平缓 | 陡峭 |
| 质量保证 | 不确定 | 确定 |

**为什么选择强制**：

```
AI 代理像人类一样会走捷径。

"这只是个小改动"
"这次不需要测试"
"我会回来加测试的"

一旦允许例外，例外就会变成常态。
```

**using-superpowers 的绝对语言**：

```markdown
<EXTREMELY-IMPORTANT>
If you think there is even a 1% chance a skill might apply,
you ABSOLUTELY MUST invoke the skill.

IF A SKILL APPLIES TO YOUR TASK, YOU DO NOT HAVE A CHOICE.
YOU MUST USE IT.

This is not negotiable. This is not optional.
You cannot rationalize your way out of this.
</EXTREMELY-IMPORTANT>
```

这种强硬语气是因为温和的建议会被 AI 合理化掉。

### 决策 2：技能自动触发 vs 手动调用

**选择**：自动触发

**权衡**：

| 方面 | 手动调用 | 自动触发 |
|------|----------|----------|
| 控制感 | 高 | 低 |
| 遗忘风险 | 高 | 无 |
| 学习负担 | 高 | 低 |
| 适用性 | 特定场景 | 通用场景 |

**语义搜索机制**：

```
技能描述：AI 通过语义搜索匹配用户意图

用户说："帮我调试这个 crash"
        ↓
AI 搜索："debug", "crash", "error"
        ↓
匹配：systematic-debugging 技能
        ↓
自动触发
```

### 决策 3：子代理驱动 vs 单代理

**选择**：子代理驱动

**权衡**：

| 方面 | 单代理 | 子代理驱动 |
|------|--------|-----------|
| 上下文连续性 | 高 | 低 |
| 隔离性 | 低 | 高 |
| 并行能力 | 无 | 有 |
| 审查独立性 | 低 | 高 |

**子代理的优势**：

```
单代理问题：
- 容易陷入自己的思路
- 审查自己不够客观
- 上下文污染

子代理解决方案：
- 每个任务有独立的"大脑"
- 审查者没有实现者的偏见
- 失败不影响其他任务
```

### 决策 4：两阶段审查

**选择**：规格合规 → 代码质量

**为什么分开**：

```
常见失败模式：
代码写得很好，但不匹配需求。

单一审查容易忽略这个问题，因为审查者被代码质量分散注意力。

两阶段审查：
1. 怀疑态度：这真的实现了需求吗？
2. 质量检查：代码写得好吗？
```

**审查角色**：

```
规格审查者：怀疑的检察官
- "让我看代码，不听你的报告"
- "需求要求 X，代码只有 Y"
- "这里有计划中没有的东西"

质量审查者：经验丰富的工程师
- "这个函数太大"
- "测试覆盖不足"
- "命名不清晰"
```

## 为什么这些原则重要

### AI 代理的特殊性

```
人类开发者：
- 有长期经验
- 有团队文化约束
- 有职业声誉关注

AI 代理：
- 每次会话"重新开始"
- 没有内在的长期记忆
- 倾向于走捷径
```

**Superpowers 为 AI 提供**：

1. **外部记忆**：技能作为持久知识
2. **强制约束**：防止走捷径
3. **最佳实践**：累积人类经验

### 质量的代价

```
快速但脆弱的代码：
- 写得快
- 调试时间长
- 维护成本高
- 容易出问题

缓慢但坚固的代码：
- 写得慢（相对）
- 调试时间短
- 维护成本低
- 可靠性强
```

**Superpowers 选择后者**：

```
短期：可能感觉"啰嗦"
长期：节省大量时间和头痛
```

## 反模式：常见错误

### 错误 1："我太忙了，没有时间"

**现实**：

```
"没有时间做测试" = "有时间调试"

没有 TDD：
编写代码：1 小时
调试：3 小时
修复：2 小时
总计：6 小时

有 TDD：
测试：30 分钟
编码：1.5 小时
总计：2 小时
```

### 错误 2："这是特殊情况"

**现实**：

```
特殊情况很快变成常规情况。

第一次："紧急情况"
第二次："类似的紧急情况"
第十次："这就是我们做事的方式"
```

### 错误 3："我知道我在做什么"

**现实**：

```
专业知识不等于完美无缺。

最优秀的开发者也会：
- 忘记边界情况
- 引入回归
- 过度设计
- 走捷径

Superpowers 帮助所有人，包括专家。
```

## 应用到其他项目

### 如何应用这些原则

1. **识别你的工作流**
   - 你的团队如何开发功能？
   - 哪些步骤是隐式的？
   - 哪些步骤经常被跳过？

2. **使工作流显式化**
   - 写下每个步骤
   - 定义完成标准
   - 设置检查点

3. **强制执行**
   - 使用自动化工具
   - 代码审查检查清单
   - CI/CD 门禁

4. **持续改进**
   - 收集反馈
   - 测量结果
   - 调整流程

## 结论

Superpowers 的设计哲学很简单：

```
做正确的事，而不是快速的事。

速度来自于减少返工，
来自于减少调试，
来自于减少意外。

这些来自于正确的过程。
```

这不是关于限制创造力，
而是关于为创造力提供可靠的基础。
