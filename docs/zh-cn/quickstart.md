## 一、什么是 Superpowers？

### 问题背景

使用 AI 编码代理（如 Claude Code）时，你是否遇到过以下情况：

- AI 直接开始写代码，但理解的需求不对
- 代码写完后发现结构需要大改
- 没有测试，不知道改动是否破坏了现有功能
- 修复一个问题，引入两个新问题

### 解决方案

**Superpowers** 是一套为 AI 编码代理设计的"工作流超能力"：

```
传统方式：需求 → 代码 → 调试 → 修改 → 再调试...

Superpowers：需求澄清 → 设计验证 → 计划分解 → TDD 开发 → 代码审查 → 完成
```

### 核心价值

| 价值 | 说明 |
|------|------|
| **质量优先** | 强制测试驱动开发，代码有保障 |
| **思路清晰** | 先设计再编码，避免返工 |
| **可追溯** | 每个步骤都有文档和验证 |
| **自动化** | 技能自动触发，无需手动调用 |

## 二、安装与验证

### 步骤 1：安装插件

在 Claude Code 中执行：

```bash
# 注册插件市场
/plugin marketplace add obra/superpowers-marketplace

# 安装 Superpowers
/plugin install superpowers@superpowers-marketplace
```

### 步骤 2：验证安装

启动新会话，输入类似请求：

> "帮我规划一个用户认证功能"

或

> "让我看看这个 bug 是怎么回"

如果 Claude 自动显示类似"Using brainstorming to refine your idea"或"Using systematic-debugging to diagnose the issue"的消息，说明安装成功。

- [ ] 插件已安装

## 三、基础工作流：七步法

Superpowers 将软件开发分解为七个明确的阶段。让我们用一个简单例子来理解：

**示例任务**：为一个 TODO 应用添加"任务优先级"功能

### 阶段 1：头脑风暴（Brainstorming）

**触发时机**：任何创造性工作之前

**发生什么**：
- AI 会先询问你想要实现什么
- 一次只问一个问题，不会信息过载
- 探索不同的实现方案
- 分块展示设计供你验证

**示例对话**：
```
AI: 你希望优先级有哪些级别？

你: 高、中、低三个级别就行

AI: 高优先级任务在界面上应该如何突出显示？
    A) 红色标记
    B) 置顶显示
    C) 红色+置顶
    D) 其他方式

你: 选 C，红色背景并且排在列表最前面
```

### 阶段 2：Git 工作树（Git Worktrees）

**触发时机**：设计批准后

**发生什么**：
- 创建独立的开发工作空间
- 确保主分支保持干净
- 新分支有独立的文件目录

**为什么重要**：
```
without worktree:
  主项目/
    ├── feature-1/ (正在开发)
    ├── feature-2/ (文件混乱)
    └── feature-3/ (难以切换)

with worktree:
  主项目/       (主分支，干净)
  worktree-1/   (独立工作空间)
  worktree-2/   (独立工作空间)
```

### 阶段 3：编写计划（Writing Plans）

**触发时机**：设计完成后

**发生什么**：
- 将功能分解为 2-5 分钟的小任务
- 每个任务都有明确的文件路径和验证步骤
- AI 估算每个任务的复杂度

**示例计划**：
```markdown
## 任务 1：添加优先级字段到数据模型
- 文件：src/types/todo.ts
- 操作：添加 `priority: 'high' | 'medium' | 'low'` 字段
- 验证：TypeScript 编译通过

## 任务 2：更新存储接口
- 文件：src/storage/todo-storage.ts
- 操作：修改保存/读取方法支持优先级
- 验证：现有测试通过

## 任务 3：实现排序逻辑
- 文件：src/utils/sort-todos.ts
- 操作：创建按优先级排序的函数
- 验证：单元测试覆盖三种优先级

...（更多任务）
```

### 阶段 4：子代理驱动开发（Subagent-Driven Development）

**触发时机**：执行计划时

**发生什么**：
- AI 为每个任务启动"子代理"
- 子代理完成任务后，主代理进行审查
- 两阶段审查：规格合规性 → 代码质量

**工作流程**：
```
主代理 → 分派任务 → 子代理 A
                    ↓
                  完成任务
                    ↓
                提交工作
                    ↓
        主代理审查 ← 规格合规审查 ← 代码质量审查
                    ↓
              通过/要求修改
                    ↓
              继续下一个任务
```

### 阶段 5：测试驱动开发（Test-Driven Development）

**触发时机**：编写任何实现代码之前

**核心循环**：RED → GREEN → REFACTOR

```
RED    ──→ 写一个失败的测试
         │
         ↓ 观察它按预期方式失败
GREEN   ──→ 编写最少的代码让测试通过
         │
         ↓ 确认测试通过
REFACTOR─→ 清理代码
         │
         ↓ 确保测试仍然通过
         └─→ 重复下一个测试
```

**铁律**：
```
没有失败的测试，就没有生产代码。
```

### 阶段 6：代码审查（Code Review）

**触发时机**：每个任务完成之间

**发生什么**：
- 对照计划检查实现
- 按严重程度分类问题
- 关键问题阻止进度

**问题分类**：
| 级别 | 说明 | 示例 |
|------|------|------|
| 阻塞 | 必须修复才能继续 | 缺少计划中的功能 |
| 重要 | 应该修复 | 测试覆盖不足 |
| 建议 | 可以改进 | 变量命名不清晰 |

### 阶段 7：完成分支（Finishing Branch）

**触发时机**：所有任务完成

**发生什么**：
- 运行完整测试套件
- 清理工作树
- 呈现选项：合并 / PR / 保留 / 丢弃

## 四、第一次实战

### 练习任务

让我们用 Superpowers 实现一个简单的功能：

> **任务**：为一个计数器添加"重置"按钮

1. 在你的项目中，告诉 Claude：
   ```
   我想给计数器组件添加一个重置按钮
   ```

2. 观察 AI 的行为：
   - [ ] 它是否先问问题而不是直接写代码？
   - [ ] 它是否创建或使用工作树？
   - [ ] 它是否先写测试？

3. 跟随 AI 引导的流程完成功能

### 预期结果

完成后，你应该有：

- [ ] 一个设计文档（在 `docs/plans/` 目录）
- [ ] 一个实施计划（在 `docs/plans/` 目录）
- [ ] 通过的测试
- [ ] 干净的 Git 历史

## 五、常见问题

### Q1：技能没有自动触发怎么办？

**检查**：
1. 确认已安装最新版本的插件
2. 启动新会话（技能在会话开始时加载）
3. 尝试明确提及："使用 Superpowers 来..."

### Q2：可以跳过某些阶段吗？

**不推荐**。Superpowers 的设计是经过验证的最佳实践组合。跳过阶段会失去质量保证。

### Q3：这会让开发变慢吗？

**短期**：首次使用可能感觉"啰嗦"

**长期**：减少返工、调试和修复时间，整体更快

### Q4：支持哪些编程语言？

**所有语言**。Superpowers 是工作流系统，与语言无关。
